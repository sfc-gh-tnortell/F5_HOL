3. Module 1: Support & Telemetry Agent

In this module you'll build a Cortex Agent specialized for customer support and telemetry analysis. The goal: discover which customers have telemetry signals that correlate with open support cases.

Step 1: Analyze SQL Query Patterns CoCo

We have 60 historical SQL queries written by different personas (Support Analyst, Data Engineer, Sales Ops, CSM, Executive). Analyze them to identify the most important patterns and distill them into 5 verified queries for a Cortex Skill.

Open Account_Prep/setup/12_query_repository_support_cs.sql and use the following prompt:

Analyze the 60 SQL queries in this file. These were written by different personas 
to answer support and telemetry questions. I need you to:

1. Identify the key dimensions, metrics, and join patterns used across all queries
2. Group queries by intent (case analysis, telemetry correlation, SLA tracking, etc.)
3. Find inconsistencies where different queries calculate the same metric differently 
   and recommend which calculation to standardize on
4. Distill the top 5 most impactful queries that cover the broadest analytical needs
5. For each of the 5 queries, write it as a clean SELECT statement with a comment 
   explaining the business question it answers and which persona asks it

These will be used as verified query representations (VQRs) when building 
a semantic view. Write them as plain SELECT statements, not functions.

Also create a reusable Cortex Code skill (SKILL.md) that captures this 
analysis workflow so I can repeat it on a different query repository in 
another domain. Install it to ~/.snowflake/cortex/skills/query-analysis/

Expected Output:

  • HOL/verified_queries.sql — 5 SELECT statements, each with business question + persona header
  • ~/.snowflake/cortex/skills/query-analysis/SKILL.md — Reusable Cortex Code skill for any domain
  • Analysis summary: table frequency, join patterns, identified inconsistencies with recommendations

Step 2: Create Support/Telemetry Semantic View

Using the verified queries from Step 1, create a semantic view. Choose one of the two options below:

Option A: Cortex Code CoCo

Open the verified queries file generated in Step 1 and use the following prompt:

Using the verified queries in HOL/verified_queries.sql, create a semantic view 
for support and telemetry analysis. The queries show the tables, joins, 
dimensions, and metrics that matter most.

Create it as: F5_PROD.FINAL.F5_SUPPORT_TELEMETRY_SEMANTIC_VIEW

Include the verified queries as VQRs in the semantic view definition.

Option B: Snowsight UI (Semantic Auto-Generator) Manual

  1. Navigate to AI & ML → Cortex Analyst → Semantic Views
  2. Click + Create Semantic View
  3. Select Generate from verified queries
  4. Upload or paste the contents of HOL/verified_queries.sql
  5. The auto-generator will infer tables, relationships, dimensions, metrics, and facts from the SQL
  6. Review the generated semantic view structure:
    • Confirm tables: DIM_SUPPORT_CASE, FACT_SUPPORT_CASE, COL_XC_TELEMETRY, COL_XC_PRODUCT_HEALTHSCORE, DIM_CUST_ACCT_SFDC
    • Confirm relationships are joined on SFDCF5_ACCT_ID and SUPPORT_CASE_ID
    • Confirm metrics include: open_case_count, avg_resolution_hours, sla_breach_rate, avg_bot_transactions
  7. Set the name to F5_PROD.FINAL.F5_SUPPORT_TELEMETRY_SEMANTIC_VIEW
  8. Click Create
Key Insight: The telemetry signals (bot-defense, waf, capacity, load-balancer, dns, performance) map directly to support case categories. This is the correlation the agent will discover — but it's not explicit in the schema.

Step 4: Connect Jira via MCP Manual

Set up the Atlassian MCP connector so the agent can read and create Jira tickets.

Requires ACCOUNTADMIN role. The API integration and MCP server are account-level objects.
  1. Run the following SQL in a Snowsight worksheet: 
    USE ROLE ACCOUNTADMIN;
USE SCHEMA F5_PROD.PUBLIC;

-- Create the API integration using dynamic client registration
CREATE API INTEGRATION jira_mcp_api_integration
  API_PROVIDER = external_mcp
  API_ALLOWED_PREFIXES = ('https://mcp.atlassian.com/v1/mcp')
  API_USER_AUTHENTICATION = (
    TYPE=OAUTH_DYNAMIC_CLIENT,
    OAUTH_RESOURCE_URL='https://mcp.atlassian.com/v1/mcp'
  )
  ENABLED = TRUE;

-- Create the external MCP server
CREATE EXTERNAL MCP SERVER atlassian_mcp_server
  WITH DISPLAY_NAME = 'Atlassian (Jira & Confluence)'
  URL='https://mcp.atlassian.com/v1/mcp'
  API_INTEGRATION = jira_mcp_api_integration;

-- Grant access to all roles
GRANT USAGE ON EXTERNAL MCP SERVER atlassian_mcp_server TO ROLE PUBLIC;
GRANT USAGE ON INTEGRATION jira_mcp_api_integration TO ROLE PUBLIC;
  2. Add the MCP tool to your agent:
    • Navigate to AI & ML → CoWork
    • Find F5_SUPPORT_AGENT → click Edit
    • Click + Add Tool → select MCP
    • Select atlassian_mcp_server
    • Click Save
  3. Authenticate: The first time you use the MCP tool, you'll be prompted to sign in to Atlassian and authorize access to your Jira workspace (f5snowhol.atlassian.net)
Jira Project: Tickets are in the KAN project. Case numbers in DIM_SUPPORT_CASE.SUPPORT_CASE_NUM match Jira keys (e.g., KAN-123).

4. Module 2: Churn Propensity Model

Build a predictive model that identifies accounts likely to churn based on telemetry trends, support case volume, and health scores.

Step 1: Build Churn Propensity Model CoCo

Build a telemetry-driven churn propensity model using the F5 data. 

Data sources:
- COL_XC_TELEMETRY (12 months daily telemetry per account)
- DIM_SUPPORT_CASE (support case volume and categories)
- FACT_SUPPORT_CASE (resolution times, SLA breaches)
- COL_XC_PRODUCT_HEALTHSCORE (CONSUMPTION_PATTERN = 'Declining' is the target)

The model should:
1. Predict which accounts have CONSUMPTION_PATTERN = 'Declining' 
2. Use Snowpark ML or Cortex ML (classification)
3. Register the model in the Snowflake Model Registry
4. Output predictions as a table the agent can query

Target variable: CONSUMPTION_PATTERN = 'Declining' (binary classification)
Features: derived from telemetry trends, support patterns, health metrics

Step 2: Feature Engineering CoCo

Create features for the churn model from our telemetry and support data:

1. Telemetry trend slopes (over last 90 days):
   - HTTP load balancer count trend (increasing/flat/declining)
   - WAF usage trend
   - Bot transaction volume trend
   - Endpoint count trend

2. Support case patterns:
   - Total open cases in last 60 days
   - Average case resolution time
   - SLA breach count
   - Escalation count

3. Health score signals:
   - SKU utilization percentage
   - Current consumption pattern

4. Combine into a feature table: F5_PROD.RAW.CHURN_FEATURES

Then train the classification model predicting 'Declining' consumption pattern.
Register it and create a predictions table: F5_PROD.FINAL.CHURN_PREDICTIONS
Tip: Use REGR_SLOPE in Snowflake SQL for calculating trend slopes over time windows. A negative slope on load balancer count + high support case volume is a strong churn signal.

Step 3: Add Model to Agent Manual

  1. Add the F5_PROD.FINAL.CHURN_PREDICTIONS table to the support/telemetry semantic view
  2. Add dimensions: CHURN_RISK_SCORE, PREDICTED_PATTERN, TOP_RISK_FACTOR
  3. Add metric: high_risk_accounts (count where risk score > 0.7)
  4. Update the agent system prompt to include churn awareness: 
    When identifying at-risk accounts, also check their churn propensity score.
Accounts with declining telemetry trends + open support cases + high churn score 
need immediate attention. Recommend proactive outreach before they file more cases.

Step 4: Proactive Investigation Manual

Test the enhanced agent with predictive questions:

#QuestionExpected Insight
1"Which accounts have a high churn risk score but no open support cases yet?"These are accounts with declining telemetry that haven't filed cases — proactive outreach targets
2"Show me accounts where telemetry is declining AND they had cases resolved in the last 3 months."Previously-resolved issues may be recurring — check if telemetry confirms
3"Can we predict which accounts will file support cases next month based on current telemetry trends?"Model identifies accounts with early-stage degradation patterns
4"Create a Jira ticket for any high-risk account that doesn't already have an open case."Proactive case creation via MCP based on model predictions
Success Criteria: The model identifies accounts with declining patterns before they escalate. Combined with the telemetry-support correlation from Module 1, the agent can now provide proactive recommendations rather than reactive case management.
